# 業務ダッシュボード 技術要件書

AI エージェントおよび開発チームが実装に着手するための技術仕様を定義する。機能の「何を作るか」は `要件定義.md`、UI の見た目・操作は `デザイン要件.md` を参照する。

---

## 1. 開発スタック

### 1.1 一覧

| カテゴリ | 選定技術 | バージョン / 備考 |
|----------|-----------|-------------------|
| フロントエンド | Next.js | v14 以降、**App Router**、TypeScript 必須 |
| UI | Tailwind CSS + shadcn/ui | コンポーネントは shadcn 準拠で拡張 |
| アイコン | Lucide React | 統一して使用 |
| BaaS / バックエンド | Supabase | Auth、PostgreSQL、Storage、Edge Functions |
| クライアント | `@supabase/supabase-js` | ブラウザ・サーバー両方から利用可 |
| 型 | Supabase CLI 生成型 | `database.types.ts` 等の自動生成を推奨 |
| AI | OpenAI API / Anthropic API | モデル例: GPT-4o、Claude 3.5 Sonnet（プロジェクトで固定） |
| AI 呼び出し経路 | **Supabase Edge Functions** | API キーはクライアントに露出させない |
| PDF | react-pdf（クライアント/サーバー方針は実装時に決定） | 複雑レイアウト時は Puppeteer 等の併用を検討 |
| OCR / 画像理解 | OpenAI **Vision** API | パスポート・在留カードの抽出に使用 |
| ベクトル検索 | **pgvector**（Supabase） | マッチング・ナレッジ用 |

### 1.2 実行環境

- Node.js: LTS（プロジェクト README に固定版を記載すること）
- デプロイ: Vercel 等、Next.js と相性の良いホスティングを想定
- 環境変数: `NEXT_PUBLIC_*` は公開可能な値のみ。秘密鍵はサーバー・Edge のみ

## 2. アプリケーション構成（ルーティング目安）

| パス例 | 役割 |
|--------|------|
| `/` または `/dashboard` | 3×2（＋α）ダッシュボード |
| `/candidates` | 候補者一覧 |
| `/candidates/[id]` | 候補者詳細（Tabs） |
| `/clients`, `/orders` 等 | 案件・クライアント（段階的に追加） |
| `/documents` | 書類・OCR ジョブ状況 |
| `/settings` | 組織・ユーザー設定 |

**Drill-down 原則**: カードクリック → 一覧 → `[id]` 詳細。詳細は shadcn `Tabs` で「基本情報 / 書類 / 履歴 / AI分析」。

## 3. データベース設計（Supabase / PostgreSQL）

### 3.1 共通方針

- 全テーブルで **Row Level Security (RLS)** を有効化する
- **テナント境界**: `organization_id`（または同等）でデータを分離し、JWT のクレームと照合する
- タイムスタンプ: `created_at`, `updated_at` を標準化
- ソフトデリートが必要なエンティティは `deleted_at` を検討

### 3.2 テーブル定義（論理）

### 3.3 インデックス・検索

- 頻繁なフィルタ: `organization_id`, `candidates.status`, `placements.status`
- ベクトル: `embedding` に ivfflat または supabase 推奨のインデックス

### 3.4 Storage バケット

| バケット例 | 用途 | ポリシー |
|------------|------|----------|
| passports | パスポート画像 | 認証ユーザーのみ自組織パス下 |
| generated-pdfs | 生成 PDF | 同上 |

**推奨**: Storage アップロード完了後、**Edge Function**（Webhook / Database trigger 経由）で OCR パイプラインを起動。

## 4. 認証・認可

- Supabase Auth（メール、将来 SSO を拡張可能な設計）
- RLS ポリシー例: `auth.uid()` に紐づく `profiles.organization_id` と行の `organization_id` が一致する場合のみ SELECT/INSERT/UPDATE
- サービスロールキーは **Edge Functions のみ**（クライアント禁止）

---

## 5. AI・自動化フロー（実装要件）

### 5.1 OCR → 候補者更新 → PDF

1. クライアントが画像を Storage にアップロード（署名付き URL または RLS 付きバケット）
2. Edge Function 起動（Storage Event または DB / キュー）
3. OpenAI Vision で JSON スキーマに沿った抽出（プロンプトでフィールド固定）
4. `candidates` の該当レコードを更新（`ocr_passport_json` 等）
5. react-pdf（またはサーバー PDF）でテンプレートにマッピングし `documents` + Storage に保存

**エラー時**: 失敗ステータスを `documents` またはジョブテーブルに記録し、UI で再試行可能にする。

### 5.2 マッチング・スコアリング

- `orders.requirements` と `candidates` のテキスト特徴を **embedding** 化し類似度算出
- 過去 `placements` で `早期終了` 等のネガティブ事例を重み付け参照（ルール＋LLM のハイブリッド可）
- Edge Function で「マッチ率（%）」と「理由テキスト」を生成し、キャッシュテーブルまたは一時レスポンスで返す

### 5.3 ナレッジ Q&A

- ドキュメントをチャンク化し `knowledge_base` に embedding 保存
- ユーザークエリを embedding → 近傍検索 → LLM で回答（引用チャンク ID を返すと信頼性向上）

## 6. フロントエンド実装指針

- **データ取得**: Server Components を優先し、機密トークンはサーバー側のみ
- **フォーム**: React Hook Form + Zod（shadcn 定番）
- **一覧**: TanStack Table 等は規模に応じて導入
- **リアルタイム**: 必要箇所のみ Supabase Realtime（ダッシュボードのキー数値）

---

## 7. 品質・運用

| 項目 | 要件 |
|------|------|
| 型安全 | DB 型生成、API レスポンスは zod で検証推奨 |
| ログ | Edge Functions に構造化ログ（個人情報はマスク） |
| レート制限 | 公開 API は Edge 側で制限 |
| 秘密情報 | OpenAI/Anthropic キーは Supabase Secrets |
